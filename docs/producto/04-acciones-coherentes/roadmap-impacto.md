# Roadmap de impacto

## ¿Para qué sirve este documento?

Un **roadmap basado en oportunidades**, no en features. Cada entrada describe **qué oportunidad atacamos**, **qué impacto esperamos en el cliente** y **qué hipótesis validamos**. La solución concreta puede cambiar — lo que se compromete es el outcome.

## Diferencia con un roadmap tradicional

| Roadmap tradicional | Roadmap de impacto |
|---------------------|---------------------|
| Lista de features con fechas | Lista de oportunidades con impacto esperado |
| "Vamos a hacer X" | "Vamos a resolver el problema Y midiendo Z" |
| Compromiso con la solución | Compromiso con el outcome |
| Discusión sobre features | Discusión sobre hipótesis |

## Preguntas clave que debe responder

1. ¿Qué **oportunidades** atacamos en cada trimestre?
2. ¿Qué **impacto esperamos** en el cliente?
3. ¿Qué **hipótesis** validamos?
4. ¿Qué **soluciones** consideramos (sin casarnos)?
5. ¿Qué **riesgos** vemos?

## Recomendación

- Detalla **a fondo Q1**, esboza Q2-Q3, marca Q4 como "exploratorio".
- Cada trimestre tiene **1-3 oportunidades core**, no más.
- Si una oportunidad no tiene assumption tests claros, no es momento de meterla al roadmap.

# Principios para gestionar el roadmap

1. **El roadmap es vivo.** Se revisa al menos una vez por trimestre.
2. **Los compromisos son outcomes, no features.** Si descubrimos que otra solución sirve mejor a la oportunidad, cambiamos la solución.
3. **Si una oportunidad no se valida, se reemplaza.** No vale "ya casi" — la siguiente.
4. **Antes de bajar a Q+1, revisar:** ¿el aprendizaje del trimestre cambia algo? Actualizar el OST y reflejarlo aquí.
5. **No incluir features sueltas.** Si algo no sirve a una oportunidad del OST, no entra al roadmap.
